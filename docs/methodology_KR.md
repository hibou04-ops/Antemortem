## 초록

**Antemortem**은 소프트웨어 변경에 대한 *전술적* 사전 정찰 규율(discipline)이다. 코드를 작성하기 전, 계획된 변경을 *문서상에서* 스트레스 테스트한다: LLM에게 명세와 변경 영향 파일을 전달하고, 가설된 위험을 `REAL` / `GHOST` / `NEW` / `UNRESOLVED`로 분류하되 *primary-source `file:line` 인용*을 강제한다. 그 결과를 바탕으로 구현 *이전에* 명세를 수정한다.

Gary Klein이 HBR(2007)에서 제시한 **pre-mortem**과는 두 가지 면에서 다르다:
- **범위**: 프로젝트 전체가 아닌 단일 코드 변경
- **입자도(granularity)**: 팀 전략 브레인스토밍이 아닌, 실제 소스코드를 대상으로 15~30분 내 수행

이 문서는 7단계 프로토콜, 분류 분류 체계(taxonomy), 그리고 이 규율을 뒷받침하는 사례 연구 증거를 명세한다.

## 명제

Postmortem은 무언가가 *깨진 후에* 작성한다.
**Antemortem**은 *짓기 전에* 수행하는 작업이다.

LLM의 능력이 두 가지를 값싸게 만들었다:
1. **코드베이스 읽기** — 유능한 모델이 12개 파일을 스캔하고 호출 패턴을 추적해 누가 무엇을 호출하는지 정확히 식별하는 데 한 자릿수 분(分)이면 충분하다.
2. **실패 모드 열거** — 모델은 인간이 혼자 하는 것보다 더 빠르고 더 철저하게 그럴듯한 함정을 나열할 수 있다.

값싸지지 않은 것: **올바른 코드를 쓰는 것**.

따라서 AI가 이해(comprehension)와 위험 열거(risk enumeration)를 사전에 값싸게 처리할 수 있다면, 합리적인 수순은 *비싼 단계 이전에* 그것을 하는 것이지 *그동안에* 하는 것이 아니다. 이는 비싼 평가를 갖는 최적화에서 surrogate model이 작동하는 논리와 같다 — 가설과 비싼 실험 사이에 값싼 분석을 삽입한다.

## 7단계 프로토콜

각 단계는 짧다. 결과물은 하나의 문서다.

1. **변경을 진술하라**. 한 단락. 무엇을 추가/제거/리팩터하려는가? 어떤 문제를 해결하는가? 사용자에게 보이는 효과는?
2. **함정을 열거하라**. 가설된 위험을 모두 나열하라. 관대하게. 명백해 보이는 것도 포함. 각각을 라벨링: "trap" (진짜라고 생각), "worry" (확신 없음), "unknown" (이 영역은 아직 생각 못 함).
3. **계획 + 저장소를 LLM에 넘겨라**. 변경이 건드릴 파일을 읽도록 요청. 각 함정을 실제 코드에 비추어 검증하도록 요청.
4. **Primary-source 인용으로 분류하라**. 각 함정에 대해:
   - **Real**: 코드가 위험을 확인. file + line 인용. 목록 유지.
   - **Ghost**: 코드가 위험이 적용되지 않음을 보임 (예: 두려워한 함수가 재귀하지 않음, 걱정한 메서드가 루프하지 않음). file + line 인용. 목록에서 제거하되 항목은 유지 — *어떤 일이 일어나지 않았는지*에 대한 부담을 지는 증거.
   - **New**: LLM이 표면화한, 본인이 생각하지 못한 무언가. 추가하고 분류.
5. **확률을 갱신하라**. 전체 P(성공)을 추적 중이라면 새 함정 카운트로 갱신. 숫자는 거칠지만 정찰의 함의를 본인이 떠안도록 강제한다.
6. **명세를 수정하라**. 정찰이 명세 가정과 다른 필드 또는 제어 흐름이 필요함을 드러냈다면, *지금* 명세를 바꿔라. 명세가 부담스러워지기 *전에*.
7. **문서를 저장하라**. 체크인하든, 어딘가에 보관하든. 가치는 정찰 자체뿐 아니라, 미래의 본인이 구현 중 놀랐을 때 다시 읽고 *어떤 가정이 깨졌는지* 볼 수 있다는 것이다.

**4단계를 절대 건너뛰지 마라.** 인용 없이 LLM의 분류를 받아들이고 싶은 유혹이 있다. *인용이 곧 규율이다.* 인용 없이는 한 형태의 손짓을 다른 형태의 손짓과 교환했을 뿐이다.
